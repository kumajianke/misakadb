# MisakaDB 基于 MAC 限流的可行性与替代方案

## 项目概述

当前 `MisakaDB` 的服务端是基于 TCP 长连接的，自定义登录逻辑在 `command/Func.go` 中完成。登录成功后，服务端主要记录：

- `serviceContext.LoginUser`

也就是说，当前项目已经天然具备“基于应用层身份”做控制的基础，而不是必须依赖网络层标识。

## 核心结论

如果你的场景里：

- 对端地址会变化
- 可能经过 VPN
- 可能跨网段、跨 NAT、跨代理

那么“基于 MAC 做限流”通常**不可靠，也不应该作为主方案**。

原因不是实现难，而是模型本身不成立。

## 为什么 MAC 限流不可靠

### 1. 服务端通常拿不到真实客户端 MAC

MAC 是二层地址，只在本地链路里有效。只要连接经过：

- 路由器
- VPN
- NAT
- 代理

服务端看到的就不是客户端原始二层信息。

### 2. VPN 会改变你看到的网络身份

使用 VPN 后，服务端通常看到的是：

- VPN 出口 IP
- 隧道对端
- 中间网关

而不是客户端物理网卡的 MAC。

即使你能在某些同网段场景下通过 ARP 间接查到 MAC，那也只是局部网络现象，不能作为通用限流标识。

### 3. MAC 本身也不稳定

即使不考虑 VPN，MAC 也未必稳定：

- 虚拟网卡会变化
- 容器/虚拟机会变化
- 系统可启用随机 MAC
- 用户可以手工伪造 MAC

所以它更像“弱设备特征”，不是“强身份标识”。

## 对当前项目最合适的限流思路

## P0：按连接限流

这部分你已经在做了，例如最大连接数限制。对于数据库类服务，这是第一层防线。

适合限制：

- 全局最大连接数
- 单 IP 最大并发连接数
- 单用户最大并发会话数

## P0：按登录用户限流

你当前项目已经有：

- 用户系统
- 登录命令
- `serviceContext.LoginUser`

所以最适合优先落地的是：

- 每个用户每秒请求数
- 每个用户同时在线连接数
- 每个用户敏感命令速率

这是当前架构里最稳定、最可控、最容易解释的一层。

## P1：按会话 ID / 设备 ID 限流

如果你担心：

- 同一个用户换 IP
- VPN 切换出口
- 多端共享账户

那么比 MAC 更合理的做法是引入应用层设备标识，例如：

- 客户端首次初始化生成 `client_id`
- 保存到本地配置文件
- 每次登录时随请求上送

然后服务端按：

- `username`
- `client_id`
- `session_id`

做组合限流。

这比 MAC 更稳定，也更适合跨网络环境。

## P1：按证书或签名设备身份限流

如果你要更强的“设备绑定”：

- 客户端证书
- 签名密钥
- 设备注册机制

会比 MAC 更靠谱。

这是很多商业系统常见做法，因为它们依赖的是应用层信任链，而不是局域网链路层信息。

## P2：按 IP 作为辅助手段，不作为主身份

IP 可以辅助限流，但不要单独作为唯一身份，因为：

- NAT 下多人共用一个 IP
- VPN 下出口 IP 会变化
- 移动网络 IP 漂移频繁

更合理的做法是：

- 登录前：按 IP 做粗粒度防爆破
- 登录后：按 `用户 + 会话/设备` 做精细限流

## 对 MisakaDB 的推荐方案

### 第一层：未登录阶段

按来源 IP 做防暴力尝试限制：

- 每分钟登录失败次数
- 每个 IP 的新建连接速率
- 短时间内的异常命令频率

这一层只做粗过滤。

### 第二层：已登录阶段

按 `LoginUser` 做主限流：

- 每用户每秒命令数
- 每用户最大连接数
- 每用户高风险命令配额

这是最适合你当前项目的主方案。

### 第三层：设备维度

如果你需要“同一账号不同机器”区分能力，补一个应用层 `client_id`：

- 首次安装生成
- 客户端持久化
- 登录或心跳时上送

然后按：

- `LoginUser + client_id`

做更细粒度限流。

### 第四层：风险联动

对高风险场景增加附加策略：

- 新设备首次登录收紧速率
- 切换国家/网络环境时收紧速率
- 密码修改后旧会话降级或失效

## 如果你坚持要“类似 MAC 的设备维度”

建议不要直接使用真实 MAC，而是设计一个“设备指纹/设备 ID”体系。

可以参考成熟商业产品思路：

- 安装时生成随机 UUID
- 与用户绑定
- 服务端存档
- 可被管理员吊销

优点：

- 跨 VPN 仍可工作
- 跨 IP 仍可工作
- 不依赖局域网
- 兼容未来客户端升级

## 直接回答你的问题

在“对端地址可能变化、存在 VPN”的前提下，不建议做“根据 MAC 的限流”。

更合理的做法是：

1. 登录前按 IP 粗限流
2. 登录后按 `LoginUser` 主限流
3. 如果需要设备维度，再增加 `client_id`
4. 不把 MAC 作为核心身份依据

## 参考实践

### 开源项目常见做法

- 数据库和中间件通常按连接、账号、命令频率限流
- 零信任或企业级系统更倾向设备证书、设备 ID，而不是 MAC

### 商业项目常见做法

- 风控层会采集很多弱特征，MAC 最多只是其中一项参考
- 真正用于访问控制和限流的，通常仍是账号、会话、设备 ID、证书

## 下一步行动

1. 明确限流目标是“防爆破”还是“设备配额”
2. 先落地 `IP + LoginUser` 双层限流
3. 需要设备区分时再引入 `client_id`
4. 不建议把 MAC 设计成核心依赖
