# MisakaDB Windows Service Guide

## 项目概述

当前 `misakadb` 是一个普通命令行程序，入口位于 `misaka.go`。它可以直接前台运行，但还不等同于 Windows Service，因此不能仅靠把 `exe` 放入环境变量后就稳定地通过 `net start` 启动。

## 核心结论

`net start <服务名>` 启动的是已经注册到 Windows Service Control Manager 的“服务”，不是普通可执行文件。

因此，除了编译出 `exe` 并放到可访问路径以外，至少还需要完成下面几件事：

1. 把程序注册成 Windows 服务
2. 解决服务启动时的工作目录问题
3. 解决配置文件、用户文件、密钥文件的绝对路径问题
4. 处理服务停止时的优雅退出
5. 规划日志输出位置，避免只写控制台

## 结合当前项目的发现

### 1. 当前入口还是控制台程序

`misaka.go` 只是普通 `main()` 启动逻辑，没有接入 Windows SCM 的服务生命周期接口，因此它本身不能直接响应服务管理器的启动、停止、暂停控制。

### 2. 当前大量使用相对路径

项目里有多处相对路径依赖，服务模式下这是高风险点：

- `./profiles/misaka.yaml`
- `./profiles/master.mikey`
- `./profiles/user.dat`

Windows 服务的默认工作目录通常不是你的程序目录，常见会落到 `C:\Windows\System32`。这会导致服务启动后找不到配置和数据文件。

### 3. 当前缺少服务停止处理

服务被停止时，建议能优雅关闭监听 socket、结束 goroutine、释放资源。当前入口没有看到针对 Windows 服务停止事件或通用进程退出信号的清理逻辑。

### 4. 当前日志主要面向控制台

服务方式运行时，没有交互终端，控制台输出不再适合作为主要日志落点。至少需要文件日志、事件日志，或者由服务包装器接管输出。

## 你除了加环境变量，还要做什么

### 必做项

1. 注册服务

可选方式：

- 使用 `sc.exe create`
- 使用 `nssm`
- 使用 `WinSW`
- 在 Go 程序里直接实现 Windows Service 接口

对当前项目来说，最省事的落地方式通常是：

- 开发期或单机部署：优先 `nssm`
- 需要更规范的交付：优先 `WinSW`
- 需要原生能力：再做 Go 原生 Windows Service 化

2. 固定工作目录

你必须确保服务启动时能定位到项目目录，否则下面这些都会失效：

- `profiles/misaka.yaml`
- `profiles/master.mikey`
- `profiles/user.dat`

推荐做法：

- 不再依赖当前工作目录
- 启动参数显式传入 `-configs`
- 用户文件和密钥文件也改成配置驱动或基于程序目录计算绝对路径

3. 准备服务账户和权限

如果服务账户没有对应目录权限，也会启动失败。至少要保证服务账户对这些目录有读写权限：

- 配置目录
- 数据目录
- 日志目录
- `profiles` 目录

4. 设计停止行为

如果将来真的挂到 Windows 服务上，你需要明确：

- 收到停止命令后如何关闭 listener
- 如何等待活跃连接退出
- 是否设置超时强制退出

### 强烈建议项

1. 使用绝对路径配置

当前项目最现实的问题不是“能不能启动”，而是“服务启动后会不会找不到文件”。

建议把以下内容统一改成可配置绝对路径：

- `misaka.yaml`
- `master.mikey`
- `user.dat`
- 数据存储目录

2. 增加日志落盘

当前 `clilog` 主要输出到标准输出/错误。服务运行后建议至少支持：

- 文件日志
- 日志滚动
- 启动失败时的错误文件

3. 区分前台模式和服务模式

建议后续设计两种运行模式：

- `console` 模式：便于开发调试
- `service` 模式：用于 Windows SCM 启动

## 对当前项目最现实的落地方案

### 方案 A：先不改代码，使用服务包装器

最适合你当前阶段。

推荐顺序：

1. 编译 `misaka.exe`
2. 准备完整目录结构
3. 把配置文件和 `profiles` 放在固定目录
4. 用 `nssm` 或 `WinSW` 注册服务
5. 启动命令显式传入配置文件绝对路径

例如思路是：

- 可执行文件：`G:\misakadb\misaka.exe`
- 配置文件：`G:\misakadb\profiles\misaka.yaml`
- 服务启动参数：`-configs G:\misakadb\profiles\misaka.yaml`

这样可以先绕过默认工作目录不稳定的问题。

### 方案 B：改成原生 Windows 服务

更长期、更规范，但成本更高。

需要补的能力：

- 接入 Windows 服务 API
- 响应启动/停止命令
- 优雅退出
- 独立日志策略
- 明确服务安装/卸载命令

## 直接回答你的问题

不能只把 `exe` 加进环境变量。

如果你想让它能被 `net start` 启动，至少还要：

1. 先把它注册成 Windows 服务
2. 解决服务工作目录问题
3. 给启动命令传入绝对配置路径
4. 确保 `profiles`、数据目录、日志目录权限正确
5. 最好补上停止逻辑和日志落盘

- 服务启动参数强制使用绝对 `-configs`

### P1

- 去掉核心路径的相对路径依赖
- 把用户文件、密钥文件路径纳入配置
- 增加文件日志

- 增加优雅停止能力
- 后续再考虑改成原生 Windows Service

## 参考实践

### 开源项目常见做法

- 很多 Go 服务在 Windows 上优先用 `WinSW` 或 `nssm` 作为包装层，而不是第一时间自己实现服务接口
- 成熟服务通常不依赖当前工作目录，而是使用配置文件、环境变量或程序目录推导绝对路径
- 日志通常落盘或接入事件系统，而不是仅输出 stdout

### 更适合 MisakaDB 的建议

现阶段优先做“可部署”比“原生服务化”更划算。先通过包装器稳定启动，再逐步补齐：

- 绝对路径
- 日志
- 优雅退出
- 原生服务支持

## 下一步行动

1. 确定你要走 `nssm`、`WinSW`，还是原生 Windows Service
2. 统一确定安装目录，例如 `G:\misakadb`
3. 用绝对路径测试一次前台启动
4. 再注册成服务并测试 `net start`
5. 最后补日志和优雅退出
