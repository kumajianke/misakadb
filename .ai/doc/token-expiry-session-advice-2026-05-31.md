# MisakaDB 登录态与过期机制建议

## 项目概述

当前 `MisakaDB` 的登录实现位于 `command/Func.go`。用户执行 `__login` 后，服务端在当前连接上下文中记录：

- `serviceContext.LoginUser = username`

这说明当前项目使用的是“连接内会话态”而不是“独立 token 模式”。

## 关键发现

### 1. 当前不是 token 模式

从实现看，登录成功后没有：

- 签发 token
- 返回 access token / refresh token
- 记录 token 过期时间
- 校验 token 签名
- 做 token 黑名单或续期

当前只有在 [Func.go](file:///g:/project-pipline/misakadb/command/Func.go#L64-L86) 登录成功后，把用户名写入连接上下文；上下文结构位于 [ServiceConnContext.go](file:///g:/project-pipline/misakadb/network/context/ServiceConnContext.go#L13-L18)。

因此，当前更准确的描述是：

- 一条 TCP 连接 = 一个登录会话
- 连接断开 = 登录态失效

### 2. 在当前架构下，优先考虑“会话过期”而不是“token 过期”

因为现在没有独立 token，所以直接讨论 token 过期并不贴切。更贴近当前实现的问题是：

- 一个连接登录后，如果长时间不主动断开，是否应该一直保持已登录状态？
- 如果客户端长期挂着连接，是否要在一段时间后要求重新登录？

这类问题本质上是“会话过期 / 空闲过期 / 强制重认证”，不是 token 过期。

### 3. 当前确实值得考虑“过期”，但不一定要马上上 token

是否需要“类似 token 的过期”取决于你的产品定位：

#### 如果当前场景是：

- 单机部署
- 内网使用
- 连接短生命周期
- 客户端连上后做少量操作就断开

那么短期内不一定必须引入 token 过期。

#### 如果当前场景逐渐变成：

- 远程登录
- 长连接长期在线
- 多客户端共享账户
- 更强的权限隔离要求
- 需要审计登录状态

那么就应该尽快补“登录态过期机制”。

## 结合当前项目的建议

## 优先级判断

### P0：先补会话有效期，而不是直接上 token

对当前项目来说，最合理的第一步不是 JWT 或复杂 token 体系，而是给连接上下文增加这些概念：

- 登录时间
- 最近活跃时间
- 会话最大存活时长
- 空闲超时时长

然后在每次执行需要鉴权的命令前检查：

- 是否已登录
- 是否超出最大会话时长
- 是否空闲超时

这种做法和 Redis、MySQL 客户端、很多数据库控制台的思路更接近，也更符合你当前“基于 TCP 长连接”的架构。

### P1：把 `remote` 开关真正接入登录校验

当前用户模型里已经有：

- `Role`
- `Remote`

但从现有登录实现看，`__login` 只校验了用户名密码，没有把 `Remote` 是否允许远程登录纳入判断。

这比 token 过期更紧急，因为它属于直接的访问控制缺口。

### P1：考虑权限变更后的会话一致性

当前登录成功后只在上下文里记录了用户名，没有记录：

- 登录时角色快照
- 用户状态版本
- 会话创建时间

如果某用户登录后被管理员降权、禁远程、删用户，现有连接是否立刻失效，目前看没有机制保证。

这类问题在商业系统里通常通过以下方式解决：

- 每次敏感命令重新查用户状态
- 会话带版本号，用户状态变化后使旧会话失效
- 服务端集中管理会话并可主动踢下线

### P2：未来再考虑 token 化

只有在下面这些场景明显出现时，再考虑 token 才更划算：

- 需要无状态鉴权
- 需要 HTTP / 网关 / 多节点扩展
- 需要前后端分离或第三方调用
- 需要 refresh token / access token 体系

否则对当前项目来说，直接上 token 可能会带来额外复杂度，但收益不一定大。

## 直接回答问题

需要考虑“过期”这个问题，但当前更应该考虑的是“登录会话过期”，而不是严格意义上的“token 过期”。

因为你现在这段代码：

- 没有 token
- 只是把用户名挂在连接上下文里

所以最适合当前项目的做法是：

1. 先引入会话超时机制
2. 再补远程登录限制校验
3. 再补用户状态变更后的会话失效
4. 最后才考虑是否升级成 token 架构

## 推荐落地路径

### 第一阶段

- 增加 `LoginAt`
- 增加 `LastActiveAt`
- 增加 `SessionTTL`
- 增加 `IdleTimeout`

### 第二阶段

- 敏感命令执行前重新检查用户是否仍存在
- 检查 `Role`
- 检查 `Remote`

### 第三阶段

- 支持管理员踢掉在线会话
- 支持密码修改后旧会话失效
- 支持角色变更后旧会话失效

### 第四阶段

- 如果以后要做多节点或 API 网关，再评估 token / JWT / refresh token 方案

## 参考实践

### 开源项目常见思路

- 数据库类服务通常优先做“连接级会话控制”和“空闲超时”
- Web/API 类系统才更常使用 access token / refresh token
- 成熟商业系统会把“密码修改、用户禁用、角色变更”与“会话失效”联动

### 对 MisakaDB 的建议

当前项目更像数据库服务，不像典型 Web API 平台。

因此更推荐参考：

- MySQL / PostgreSQL 一类的连接会话控制
- Redis 一类的鉴权后连接使用模型

而不是一上来照搬 Web 系统的 JWT 方案。
